現在の git 差分を分析し、変更を適切な単位に分割して順番にコミットします。

## ワークフロー

### ステップ 1: 差分の把握

以下をまとめて取得します：
- `git status` — 変更ファイルの一覧（staged/unstaged/untracked）
- `git diff` — unstaged の差分
- `git diff --staged` — staged の差分
- `git log --oneline -5` — 直近のコミット（メッセージのスタイル確認）

### ステップ 2: コミット単位の設計

取得した差分をもとに、以下の観点でコミットを分割します：

**分割の基準（優先順位順）**:
1. **インフラ・設定** — 環境設定、依存パッケージ追加、DB マイグレーション
2. **型定義・共通ユーティリティ** — 型ファイル、共通 hooks、lib
3. **UI コンポーネント** — 再利用可能なコンポーネント
4. **ページ・機能単位** — 特定のページや機能に紐づく変更
5. **バグ修正・スタイル調整** — 独立した修正

**分割しない（1コミットにまとめる）ケース**:
- 変更が1つの機能に完全に紐づいており、分割すると中間状態でビルドが壊れる
- 変更ファイルが2〜3件以下で明確に1つの目的
- 型定義とその実装が密結合している

### ステップ 3: 順番にコミット

設計した順序で各コミットを実行します：

1. 対象ファイルを `git add` でステージング（`git add .` は使わず個別指定）
2. コミットメッセージを作成・実行
3. 次のコミットへ

**コミットメッセージの規則**:
- 1行目: 変更内容を端的に（50文字以内を目安）
- プレフィックスは使わない（`feat:` `fix:` 等は不要）
- 日本語・英語どちらでも可（リポジトリの既存スタイルに合わせる）
- 末尾に必ず Co-Authored-By を付与：
  ```
  Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
  ```

**コマンド例**:
```bash
git add path/to/file1.ts path/to/file2.ts
git commit -m "$(cat <<'EOF'
コミットメッセージ

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
EOF
)"
```

### ステップ 4: 完了報告

全コミット完了後、作成したコミットの一覧を報告します。

## 重要なルール

- `git add .` や `git add -A` は使わず、ファイルを個別に指定する
- `.env` など機密ファイルが含まれていたらコミットせずユーザーに警告する
- `--no-verify` や `--amend` は使わない（ユーザーから明示的に指示された場合のみ）
- zsh では `app/(main)/` のようなパスを `'app/(main)/'` とクォートする
- ユーザーへの事前確認は不要。コミット単位を提示したらそのまま実行する
