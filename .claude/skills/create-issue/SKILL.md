会話の文脈やユーザーの指示から GitHub issue を作成します。

## ワークフロー

### ステップ 1: issue 内容の整理

会話の文脈、ユーザーの指示、コードベースの状態から以下を判断します：

- **タイトル**: 簡潔で具体的な1行（日本語 or 英語、リポジトリの慣習に合わせる）
- **本文**: 背景・現状・期待する動作・再現手順など、必要な情報を構造化
- **ラベル**: リポジトリに存在するラベルから適切なものを選択（任意）

本文のテンプレート（内容に応じて調整する）：

**バグ報告の場合:**
```markdown
## 概要
[何が問題か]

## 再現手順
1. ...
2. ...

## 期待する動作
[本来どうなるべきか]

## 現在の動作
[実際にどうなっているか]
```

**機能要望の場合:**
```markdown
## 概要
[何を実現したいか]

## 背景・動機
[なぜこの機能が必要か]

## 提案する実装
[どう実装するかのアイデア（あれば）]
```

**タスク・改善の場合:**
```markdown
## 概要
[何をするか]

## 詳細
[具体的な作業内容]

## 完了条件
- [ ] ...
```

### ステップ 2: ユーザーに確認

作成する issue の内容（タイトル・本文・ラベル）をユーザーに提示し、問題ないか確認する。
ユーザーが修正を求めたら反映する。

### ステップ 3: リポジトリのラベル確認（ラベルを付ける場合）

```bash
gh label list
```

存在しないラベルは指定しない。リポジトリにラベルが設定されていない場合はラベルなしで作成する。

### ステップ 4: issue 作成

`gh issue create` コマンドで issue を作成する。

```bash
gh issue create --title "タイトル" --body "$(cat <<'EOF'
本文をここに記述

EOF
)"
```

ラベルがある場合:
```bash
gh issue create --title "タイトル" --label "bug" --body "$(cat <<'EOF'
本文をここに記述

EOF
)"
```

複数ラベル:
```bash
gh issue create --title "タイトル" --label "bug" --label "priority:high" --body "$(cat <<'EOF'
本文をここに記述

EOF
)"
```

### ステップ 5: 完了報告

作成された issue の URL をユーザーに報告する。

## 重要なルール

- issue 作成前に必ずユーザーに内容を確認する
- タイトルは具体的で検索しやすいものにする（「バグ修正」のような曖昧なタイトルは避ける）
- 本文には十分なコンテキストを含める（読み手が背景を知らなくても理解できるように）
- `gh` コマンドが使えない場合はユーザーにセットアップを案内する
- 会話の文脈が不十分な場合は、必要な情報をユーザーに質問する
