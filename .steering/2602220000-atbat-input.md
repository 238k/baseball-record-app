# 打席入力（シンプル版・投球カウントなし）

## 概要

Step 3: 投球カウントを記録せず、打席の最終結果のみ入力できる画面を実装する。
走者状況と得点も管理し、「動くもの」を最速で作ることを優先する。

## 作業内容

### 新規作成ファイル

- `hooks/useGameState.ts` — 試合状態管理フック。Supabase から at_bats, base_runners, runner_events を取得して現在のイニング・アウト・走者・スコア・打順を復元する
- `app/(main)/games/[id]/input/page.tsx` — 記録入力画面（Client Component）。useGameState を使用して状態管理し、打席結果ボタン・走者操作・スコア表示を統合する
- `components/game/ScoreBoard.tsx` — スコアボード表示コンポーネント
- `components/game/OutCount.tsx` — アウトカウント表示コンポーネント
- `components/game/RunnerDisplay.tsx` — 走者表示コンポーネント
- `components/game/AtBatInput.tsx` — 打席結果ボタン群コンポーネント

### 変更ファイル

- `app/(main)/games/actions.ts` — 打席記録・投手交代・試合終了用の Server Actions を追加

## 実装メモ

### データフロー

1. 画面読み込み時に useGameState が Supabase から game, lineups, at_bats, base_runners, runner_events を取得
2. at_bats を inning/inning_half 順に処理し、各打席の runner_events からアウト・得点・走者状況を復元
3. 打席結果ボタン → 確認ダイアログ → 走者・得点入力ダイアログ → Server Action で保存 → useGameState を再取得

### 状態復元ロジック（useGameState）

- at_bats を created_at 順で取得し、各打席のアウト数・走者状況を順番に再構築する
- 最後の at_bat 完了後の状態が「現在の状態」
- 3アウト到達時はイニング進行を反映
- 最後の打席の batting_order + 1 が次の打者（9の次は1に戻る）
- base_runners テーブルは「各打席開始時の走者スナップショット」として使用

### 走者・得点入力ダイアログのデフォルト値

仕様書に定義されたデフォルト動作テーブルに従う。ユーザーが変更可能。

### 得点記録の重要ルール

- 得点は runner_events (event_type='scored') で記録する
- 本塁打の打者自身も runner_events に scored で含める
- v_scoreboard は runner_events.scored のみで集計するため

### DB書き込み（1打席確定時）

1. at_bats に INSERT（result, rbi 含む）
2. base_runners に INSERT（この打席開始時の走者スナップショット）
3. runner_events に INSERT（得点・アウト等のイベント）

### イニング進行

- 攻撃側のアウトが3になったら攻守交代
- 表→裏、裏→次のイニング表
- home/visitor は game.is_home で判定

### 打順管理

- lineup の batting_order 1-9 が攻撃順
- team_side は攻撃中のチーム側（表=visitor, 裏=home）
- 打順は at_bats の同一 inning_half 内の最後の batting_order から次へ進む
